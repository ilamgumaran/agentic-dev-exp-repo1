# DR-001: Zero External Dependencies for Core Engine

## Status: ACCEPTED
## Date: 2026-05-27
## Decider: Ilam Gumaran

## Objective Alignment
Links to: OBJ-001 ([Build a Production-Quality Re-Ranking Library](../objectives/OBJ-001.md))
Impact on metrics: Build Duration (shorter), Test Suite Duration (shorter), Enhancement Ease Score (higher — no dependency conflicts)

## Context
The re-ranking library needs to be embeddable in any Java application. External dependencies (Lucene, Apache Commons, etc.) create version conflicts, increase JAR size, and add supply chain risk.

## Options Considered

### Option A: Zero Dependencies
- Pros: No conflicts, ~50KB JAR, no supply chain risk, forces deep understanding of algorithms
- Cons: More code to write, no ready-made tokenizers, limited semantic matching

### Option B: Lucene for Tokenization + Scoring
- Pros: Battle-tested, feature-rich, well-documented
- Cons: 10MB+ JAR, version conflicts common, heavy for a library

### Option C: Selective Small Dependencies
- Pros: Balance of convenience and size
- Cons: Each dependency is a supply chain risk; "selective" becomes "many" over time

## Decision
Option A: Zero external runtime dependencies. Test dependencies (JUnit 5) are allowed.

## Consequences
- More code to write and maintain for tokenizers and scoring algorithms
- No neural/ML-based semantic matching without external embeddings
- Users can bring their own Scorer implementations that use external libraries
- Pipeline is composable — users inject what they need
