# DR-004: Separate FusionStrategy from per-document Aggregator

## Status: ACCEPTED
## Date: 2026-05-28
## Decider: Ilam Gumaran

## Objective Alignment
Links to: OBJ-001 ([Build a Production-Quality Re-Ranking Library](../objectives/OBJ-001.md))
Impact on metrics: Enhancement Ease Score (new fusion methods fit cleanly), no impact on existing Build/Test metrics

## Context
Adding Reciprocal Rank Fusion (RRF) to support hybrid search (BM25 + vector). RRF needs each document's rank within each scorer's full result list. The existing `Aggregator` interface (`double aggregate(List<Score>)`) operates per-document and cannot compute ranks, which are a global property of the result set.

This decision was informed by research into Elasticsearch, OpenSearch, Weaviate, and Qdrant fusion methods. See `docs/human-learning/RANK_FUSION_AND_HYBRID_SEARCH.md`.

## Options Considered

### Option A: Force RRF into the Aggregator interface
- Pros: One interface
- Cons: Pollutes contract, requires smuggling ranks through Score objects or thread-local state

### Option B: New FusionStrategy interface (full score matrix)
- Pros: Clean separation; RRF + normalized + future methods fit; no regression to Aggregator
- Cons: Two combination concepts (pick exactly one per pipeline)

### Option C: Replace Aggregator entirely
- Pros: One concept
- Cons: Breaking change to spec 004; forces simple cases to handle the full matrix

## Decision
Option B. `FusionStrategy` (`double[] fuse(List<Document>, List<List<Score>>)`) is a parallel concept to `Aggregator`. A pipeline uses exactly one. Aggregator serves comparable-score per-document combination; FusionStrategy serves rank-based and normalized fusion.

## Consequences
- RRF, MinMax, and Z-score fusion fit cleanly
- Existing Aggregator path unchanged (no regression)
- Future methods (DBSF, CombMNZ) can be added without architectural change
- Users must choose aggregator XOR fusion; documented guidance provided
- Pipeline builder validates exactly-one-of
