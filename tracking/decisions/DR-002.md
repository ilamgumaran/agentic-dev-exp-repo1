# DR-002: Rust UI Communicates with Java via JNI

## Status: ACCEPTED
## Date: 2026-05-27
## Decider: Ilam Gumaran

## Objective Alignment
Links to: OBJ-001 ([Build a Production-Quality Re-Ranking Library](../objectives/OBJ-001.md))
Impact on metrics: Test Suite Duration (JNI = low overhead), Build Duration (two toolchains)

## Context
Need a Rust-based UI for testing the Java re-ranking library. Cross-language options: JNI (in-process), gRPC (network), subprocess+JSON (pipe), shared memory.

## Options Considered

### Option A: JNI via `jni` crate
- Pros: Microsecond overhead, no serialization, direct Java object access
- Cons: Complex, requires JDK, platform-specific

### Option B: Subprocess + JSON
- Pros: Simple, no JDK required at build time, portable
- Cons: Process startup overhead, serialization cost, harder to debug

## Decision
Option A: JNI. Performance measurement is a primary use case, so the bridge must not add overhead. JSON serialization at the boundary mitigates JNI complexity.

## Consequences
- Requires JDK on machine running the UI
- JNI surface kept minimal (one entry point)
- Subprocess+JSON fallback may be added later for environments without JDK
